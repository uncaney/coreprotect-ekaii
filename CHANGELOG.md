# Changelog

All notable fork-specific changes since the upstream CoreProtect baseline. Numbers all
verified by `bench/bench.py`; raw JSON for each release attached on the corresponding
forgejo release page.

## v23.2-ekaii-1.2.0 — 2026-05-02

DuckDB backend + Postgres performance hardening.

**Added**
- `database-backend: duckdb` first-class option. In-process columnar engine
  with native zonemap indices, lz4 disk compression, vectorized scans.
  Driver (~73 MB) is opt-in — drop into Paper's `libraries/`. Plugin
  falls back to SQLite with a console message if absent.
- `postgres-async-commit: true` — `SET synchronous_commit = off` per session.
  Trades a recoverable observation gap (PG crash can lose the last few
  hundred ms) for ~2× insert throughput on spinning disks. Off by default.
- Unix domain socket auto-detect for PG. junixsocket-core shaded. When host
  is loopback AND the socket file exists AND junixsocket is on the
  classpath, pgjdbc routes through AF_UNIX. Saves ~20 µs per syscall.
- `postgres-partition-interval: weekly|monthly` knob.
- `enable_partitionwise_join` / `_aggregate` ON for PG sessions via
  Hikari `connectionInitSql`.

**Changed**
- `co_block` and `co_container` on partitioned PG and on DuckDB now use
  `PRIMARY KEY (time, rowid)` instead of `(rowid, time)`. Inserts always
  land at the leading edge of the hot leaf — fixed PG-partitioned scan
  p50 from 10.7 ms (1.1.0) to **1.4 ms** at 500k rows.

**Verified at 1M rows, mixed NBT blob mix, vs SQLite:**
- DuckDB inserts: 1.7× faster (361k vs 207k rows/sec).
- DuckDB scan p50: 9.5× faster (1.97 vs 18.75 ms).
- DuckDB retention: 20× faster (1.4 s vs 27 s).
- DuckDB on-disk: 8% smaller (424 vs 460 MiB).
- PG-partitioned retention: 290× faster than SQLite (93 ms vs 27 s).

## v23.2-ekaii-1.1.0 — 2026-05-02

COPY-based inserts on PG + cheap fixes from the v1.0.0 code review.

**Added**
- `PgCopyBatchingStatement` — JDK `Proxy<PreparedStatement>` that intercepts
  `addBatch()` / `executeBatch()` on bulk INSERT statements and flushes
  through pgjdbc's `CopyManager` via `COPY FROM STDIN BINARY`. **3.1× faster
  inserts on PG** (45k → 139k rows/sec at 500k rows). Mirrors all
  parameter binds to a real underlying PreparedStatement so non-batch
  paths still work; falls back to executeBatch on any error — never
  silently corrupts. Type coverage: int, long, smallint, text, bytea,
  NULL, double.
- Adaptive `chunkLimit()` on `RetentionService`: 50,000 for SQLite (saves
  writer-lock round-trips), 5,000 for PG/MySQL.
- Bench harness: `--blob-size {small|medium|large|random}` flag.
  `pg_insert_copy()` validates the same wire protocol the Java proxy uses.

**Changed (must-fixes from review)**
- `WorldUtils.getWidIndex` and `LookupRaw`: `USE INDEX(...)` now gated on
  `Backend.MYSQL` only — was breaking every PG lookup.
- `PurgeCommand`: `OPTIMIZE LOCAL TABLE` (MySQL-only) replaced with
  `VACUUM (ANALYZE)` on PG.
- `Patch.versionCheck`: skips the patch chain entirely on PG (patch scripts
  emit MySQL-specific DDL).
- `UserStatement` / `PlayerLookup`: `LOWER("user") = LOWER(?)` on PG so
  username matching stays case-insensitive (PG default collation is
  case-sensitive, unlike MySQL/SQLite NOCASE).
- `Database.hasReturningKeys()` uses `dialect.supportsReturningClause()`
  so PG goes through the native RETURNING path.
- `RetentionService`: ReentrantLock around `runNow()`. Connection rotation
  every 10 chunks to dodge Hikari's `maxLifetime` eviction. Skip scheduling
  entirely when retention disabled.

## v23.2-ekaii-1.0.0 — 2026-05-02

Initial fork release.

**Added**
- `Dialect` abstraction (`SqliteDialect`, `MysqlDialect`, `PostgresDialect`)
  replacing 30+ scattered `Config.MYSQL` boolean checks.
- PostgreSQL backend (`database-backend: postgres`). pgjdbc 42.7.4 shaded.
  Schema with `bigint GENERATED BY DEFAULT AS IDENTITY`, `bytea` blob
  columns, lz4 compression on `meta`/`blockdata`/`metadata`/`data`
  (PG14+), BRIN(time) on hot tables (~30 KB index for 100M rows), partial
  index on `(wid, x, z, time) WHERE rolled_back = 0`.
- Time-range partitioning on `co_block` and `co_container` (weekly).
  `PartitionService.ensureUpcoming(N)` creates upcoming partitions on
  startup; retention drops stale partitions via O(1) `DROP TABLE`.
- `SqlTranslator` + `PgConnectionProxy` — JDBC-layer rewrite of MySQL
  idioms (`LIMIT n,m → LIMIT m OFFSET n`, bare `user → "user"`).
  Quote-aware: skips already-quoted identifiers and string literals.
- `RetentionService` — cron-driven (`MiniCron` 5-field UTC parser) sweep.
  Folia-aware async scheduler. Chunked DELETE on SQLite/MySQL,
  DROP-PARTITION on PG. Per-table caps prevent multi-hour first runs.
  `HumanDuration` parser: `30d`, `8w`, `6mo`, `1y2w`, etc.
- `/co retention status|enable|disable|set <duration>|run` command.

**Build infrastructure**
- Dual-target builds: master = MC 26.1.2 / JDK 25, branch `mc1.21.11` =
  MC 1.21.11 / JDK 21. Both compile to Java 11 bytecode (run on JDK 11+).
- Forgejo Actions matrix CI (`.forgejo/workflows/build.yml`).
- Reproducible `bench/bench.py` harness with Docker Postgres.
